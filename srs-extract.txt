1. 	Introduction
1.1 Purpose
The purpose of this SRS is to document the complete functional and non-functional
requirements for the Study Group Management System. This document is intended
to be used by the development team as a blueprint for system design, implementation,
and testing, and by stakeholders to understand the scope and expected functionality of
the final product. This SRS clearly defines what the Study Group Management
System platform is expected to do, covering all user interactions from secure user
authentication and profile management to the creation and organization of study
groups, the sharing of educational resources, and real-time collaboration through
integrated text and video chat features.

1.3 Project Vision and Product Scope
1.3.1 Vision Statement
For students and lifelong learners who are seeking a more interactive and
collaborative way to study, the Study Group Management System is a secure web
application that allows users to create study groups, share resources, and participate
in real-time discussions. Unlike traditional isolated study methods, our product
provides a synchronous learning environment with integrated text and video chat,
ensuring that members can collaborate efficiently and effectively from anywhere.

1.3.2 Product Scope
The Study Group Management System (SGMS) will cover the entire process of
organizing study groups and facilitating academic collaboration.
The system scope includes:
● Authentication & User Management Module: Manages user identity:
registration, login/logout, JWT token lifecycle, account status control, and
profile management.
● Group Management Module: Handles study group CRUD, member
join-request workflow, and admin-level group status control.
● Session & Attendance Management Module: Manages study session
scheduling, live video rooms (LiveKit), and member attendance tracking.
● Resource Module: Provides a per-group file library organised into folders,
with star ratings and Cloudinary-backed uploads.
● Community & Engagement Module: Enables group discussions (posts,
threaded comments, voting), in-app notifications, and content reporting
● Analytics & Administration Module: Delivers role-adaptive statistics and
charts for members, leaders, and admins.
The system scope exclude:
● LMS Integration: Direct synchronization or integration with external Learning
Management Systems (e.g., Canvas, Blackboard).
● Native Mobile Support: Development of native iOS or Android applications
(the focus remains on a responsive web-based experience).
● Payment Processing: Any monetization or payment gateway integration for
group memberships or shared resources.

2. 	Overall Description
2.1 Product Perspective
The Study Group Management System (SGMS) is a standalone web application
designed to streamline collaborative learning by providing a centralized platform for
creating study groups, managing educational resources, and multi-channel
communication. It is intended to consolidate scattered learning materials and
discussion threads into a single, organized environment. The system is independent of
existing enterprise learning management systems (ELMS) and focuses on user-driven
group dynamics.

2.4 Operating Environment
This section describes the environment in which the software will operate.
● Client Side: React 19 (SPA) + Vite 6; React Router v7; Zustand v5 + TanStack
Query v5 (state management); React Hook Form + Zod (forms); TailwindCSS
v4; Recharts v2 (charts); LiveKit components (real-time video); Axios v1
(HTTP); served via Nginx on port 80.
● Server Side: Node.js 20+ (ESM); Express 4; RESTful JSON API at /api; JWT
auth (access 15 min / refresh 7 days, hashed in DB); role-based middleware;
express-validator; Helmet + CORS + bcrypt security; Morgan logging; Prisma
Client v6 ORM; served on port 5000.
● Database: PostgreSQL 16; Prisma 6 (schema-first migrations, 20 models);
soft-delete via deletedAt pattern; hosted on Neon (cloud) or Docker
(self-hosted).
● Third-Party Services: Cloudinary (file storage, signed direct-upload); LiveKit
(real-time video rooms, server-side token generation).
Copyright © 2026 by Group 12

2.5 Design and Implementation Constraints
The project must adhere to the following constraints:
● Technology Stack: Must use Node.js, ReactJS, and Monolithic Architecture
as defined in the project scope.
● Security: All financial transactions and user data must be secured using
appropriate encryption and protocols (SSL/TLS).
● Performance: The system must be able to handle peak traffic during popular
movie releases without significant degradation in response time (detailed in
Section 6.0 Quality Attributes).
2.6 Assumptions and Dependencies
2.6.1 Assumptions
● Users have access to hardware (webcam/microphone) for video-based study
sessions.
● Database availability: PostgreSQL (Neon or Docker) is continuously available
with < 100 ms latency to the backend server.
● The internet connection between the student and the server is stable during
real-time communication.
2.6.2 Dependencies
● External Services: PostgreSQL 16 (critical — all data); Cloudinary v2 (high
— file uploads); LiveKit v2 (medium — live video only)
